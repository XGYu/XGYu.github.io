软定时器机制及GPIO实验

引言

本次实验的目的是开发一个实时操作系统的软定时器，具体要求如下：

  • 实现两种常见的定时类型：单次定时和周期定时；
  • 实现软定时器的结构体设计和初始化；
  • 实现定时器启动、消息激励的超时激励、停止和资源释放；
  • 实现多任务下的GPIO测试。

对于本次的软定时器设计，主要依托的就是VxWorks提供的硬件时钟中断机制，根据需要的定时器时长，申请对应的硬件时钟滴答资源。单词定时和周期定时的区别在于，周期定时在定时器到时后，会重置定时器的剩余时长。下面主要介绍定时器结构体的设计和API设计。

软定时器设计

对于软定时器模型的设计，主要编写了两个结构体，一个是定时器实例，另一个是定时器队列。这是因为在VxWorks中会运行多个进程，所以系统需要维护这些进程可能会使用的软定时器，这里就是用链表这种数据结构来维护，如果有新的定时器加入，则直接将其附在表头。

typedef struct SoftTimerNode {
  struct SoftTimerNode* priv;  	// 前驱指针
  struct SoftTimerNode* next;  	// 后继指针
  uint32_t timerID;         // 定时任务ID
  uint32_t tickTarget;      // 总时钟周期数
  uint32_t tickRemain;      // 剩余时钟周期数
  MSG_Q_ID msgQueueID;      // 消息队列ID
  uint8_t type;             // 定时器类型
  uint8_t isActive;         // 定时器暂停(0)或运行(1)
} SoftTimerNode;


typedef struct SoftTimer {
  uint32_t sysTickPerSec;  		// 系统每秒的时钟周期数
  SoftTimerNode* timers;      	// 定时任务双向链表
  int bgTaskID;            	// 后台任务ID
  SEM_ID mu;               	// 互斥锁
} SoftTimer;

在软定时器结点的设计中，引入总时钟周期数和剩余时钟周期数，这是定时的依据，假如一个消息队列实现消息激励，计时器类型是为了区分单词定时与周期定时。由于定时器实例是一个互斥资源，所以设置互斥锁的目的是确保同一时间只能有一个进程对计时器实例的资源进行修改。

下面是设计的定时器API，涉及到定时器初始化、运行、设置任务和撤销定时器。

//初始化软定时器
int SoftTimerInit();

/***
* 设置定时任务
* @MSG_Q_ID mqID       消息队列
* @float n             计时长度
* @uint8_t unitType    采用的计时单位，时钟滴答tick或者现实中的秒
* @uint8_t timerType   计时器类型，单次或周期
* @return              计时任务ID
***/
uint32_t SetTimerTask(MSG_Q_ID mqID, float n, uint8_t unitType, uint8_t timerType);

/***
* 删除定时器任务
* @uint32_t timerID    计时器任务ID
***/
void ClearTimerTask(uint32_t timerID);

/***
* 设置定时任务开始运行
* @uint32_t timerID 计时任务ID
***/
void StartTimerTask(uint32_t timerID);

/***
* 停止定时任务运行
* @uint32_t timerID 计时任务ID
***/
void StopTimerTask(uint32_t timerID);

//销毁软定时器
void SoftTimerKill();

实验测试

实验的主要测试方法为设置一个整形全局变量，由两个任务分别对这个变量执行累加操作，各自的任务都由定时器控制，采取周期定时器机制。定时器时长设置为每秒触发一次，所以可以看到两个任务交替执行，具体的过程可参见视频。
